# Incident: Onboarding 500 — `POST /api/clients` feiler

**Dato:** 2. mars 2026  
**Alvorlighetsgrad:** Høy (blokkerer alle nye brukere fra å fullføre onboarding)  
**Status:** Løst

---

## Symptom

Brukere som prøvde å opprette en "avst emming" under onboarding (manuell sti, steg «Oppsummering» → «Opprett alt») fikk feilmeldingen:

> Internal server error

Vercel-logg viste:

```
MAR 02 19:45:48.20  POST 500  www.revizo.ai  /api/clients
[api] Unhandled error: column "tripletex_account_id" of relation "accounts" does not exist
```

---

## Rotårsak

Drizzle ORM's `.insert().values().returning()` genererer automatisk en `RETURNING`-klausul som lister opp **alle kolonner definert i `schema.ts`**, inkludert `tripletex_account_id` som ble lagt til i Drizzle-migrering `0008_add_tripletex_integration.sql`.

Denne Drizzle-migreringen har **ikke blitt kjørt mot produksjonsdatabasen**. Produksjons-DB mangler derfor kolonnen, og Postgres kaster:

```
ERROR: column "tripletex_account_id" of relation "accounts" does not exist
```

Feilen ble ikke fanget av noen `try/catch` i `POST /api/clients`, og propag erte til `withTenant`-wrapperen som returnerte den generiske `500 Internal Server Error`.

### Samme mønster hadde allerede blitt fikset for `companies`

Nøyaktig samme bug fantes i `POST /api/companies` (Drizzle inkluderte `tripletex_company_id` i RETURNING, som heller ikke eksisterte i prod-DB). Dette ble fikset i commit `746882a` (02.03.2026) ved å bytte til raw SQL med eksplisitte kolonnenavn. `api/clients` ble ikke oppdatert på samme tidspunkt.

### Hvorfor var det vanskelig å finne

Supabase MCP-tilkoblingen i Cursor er koblet til **dev-prosjektet** (Project Opus), ikke produksjonsprosjektet. Alle direkteforespørsler mot databasen via MCP viste at kolonnen `tripletex_account_id` eksisterte — fordi Drizzle-migreringen er kjørt i dev. Prod er et separat Supabase-prosjekt der migreringen ikke er kjørt.

---

## Sekundærfunn og bi-feil

Under feilsøkingen ble det funnet to tilleggsfeil i `step-configure-erp.tsx` (ERP-onboarding-stien):

1. **Manglende `credentials: "include"`** på kall til `/api/companies` og `/api/clients` — uten dette videresendes ikke sesjonskaker, og Clerk kan ikke identifisere brukeren.
2. **Dobbel `.json()`-lesing** på company-responsen — response-body er en engangs-strøm; det andre `await res.json()` ville returnert en tom/ødelagt verdi.

## Løsning

### `src/app/api/clients/route.ts`

| Endring | Begrunnelse |
|---|---|
| Byttet Drizzle `.insert().returning()` til raw SQL med eksplisitte kolonner | Unngår at Drizzle inkluderer ukjente/ikke-migrerte kolonner i RETURNING |
| Pakket alle tre INSERT-operasjoner i `db.transaction()` | Atomisitet — hvis client-INSERT feiler etter at accounts er opprettet, rulles alt tilbake (ingen foreldreløse `accounts`-rader) |
| Lagt til `try/catch` med spesifikke feilmeldinger | Constraint-brudd → 400, tillatelsesavslag → 403, annet → 500 med norsk feilmelding |

```typescript
// FØR — krasjer i prod hvis kolonne mangler i DB:
const [account1] = await db.insert(accounts).values({...}).returning();

// ETTER — eksplisitt SQL, kun base-kolonner:
const acct1Rows = await tx.execute<{ id: string }>(
  sql`INSERT INTO accounts (company_id, account_number, name, account_type)
      VALUES (${companyId}, ${...}, ${...}, ${...})
      RETURNING id`
);
```

### `src/components/onboarding/step-configure-erp.tsx`

- Lagt til `credentials: "include"` på begge API-kall
- Rettet dobbel `.json()`-lesing til ett kall med `.catch(() => ({}))`
